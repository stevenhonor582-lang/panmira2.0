# 严重等级定义 (P0 / P1 / P2 / P3)

> 守静在 Step 1 分级时加载本文档。每个事故必须明确标注严重级别。

## 严重度总览

| 级别 | 名称 | 响应时间 | 影响范围 | 决策权 |
|------|------|---------|---------|--------|
| P0 | 阻断 (Critical) | 5min | 全站不可用 / 资金损失 | 立即响应+通知所有人 |
| P1 | 严重 (High) | 15min | 核心功能受损 / 部分用户不可用 | 通知 oncall + 执行 runbook |
| P2 | 中等 (Medium) | 1h | 非核心功能受损 / 性能下降 | 标准流程 + 排期修复 |
| P3 | 轻微 (Low) | 24h | 体验问题 / 单次告警 | 加入 backlog |

## P0 — 阻断 (Critical)

**定义**：必须立即响应的全站性故障。

**典型场景**：
- 全站 5xx 飙升（错误率 > 50%）
- 关键服务完全 down（health check 失败）
- 数据泄露 / 安全入侵
- 支付 / 资金链路故障
- 数据库主库不可用

**响应要求**：
- 5min 内开战时频道 + @所有人
- 自动执行降级方案
- 通知升级链全部节点
- 每 5min 同步进展

## P1 — 严重 (High)

**定义**：核心功能受损，但服务整体可用。

**典型场景**：
- 部分接口 5xx 错误率 5%-50%
- 单服务 down（如 mahubot-portal）但核心 API 正常
- 错误日志激增（但监控指标未到 P0 阈值）
- 用户报告"某功能不可用"且影响超过 100 用户

**响应要求**：
- 15min 内第一进展
- 通知 oncall + 升级链第 1 层
- 执行对应 runbook
- 每 15min 同步进展

## P2 — 中等 (Medium)

**定义**：非核心功能受损或性能下降。

**典型场景**：
- 非核心接口错误率 < 5%
- 性能下降但未到阈值
- 缓存命中率下降
- 单个用户报告的功能异常（未扩散）

**响应要求**：
- 1h 内响应
- 通知 oncall（异步）
- 排期处理，不开战时频道
- 加入 P2 任务池

## P3 — 轻微 (Low)

**定义**：体验问题或单次告警。

**典型场景**：
- 单次 5xx 后自动恢复
- 慢请求（< P99 × 2）
- 监控阈值抖动
- 内部工具问题

**响应要求**：
- 24h 内排期
- 加入 backlog
- 不通知 oncall（除非连续触发）

## 分级规则（自动）

1. **关键词命中**（最高优先级）：
   - "5xx" / "down" / "超时" / "挂了" / "崩了" → P0
   - "错误率" + 数字 > 5% → P1
   - "慢" / "卡" → P2
   - 单词"报错" → P3

2. **指标命中**：
   - error_rate > 50% → P0
   - error_rate 5%-50% → P1
   - p99_latency > 5s → P1
   - p99_latency 2s-5s → P2

3. **来源加权**：
   - 多服务同时告警 → 升一级
   - 涉及支付 / 资金 → 直接 P0
   - 来自 Sentry（用户报错）且 > 100 events → P1

4. **保守原则**：不确定时升一级（避免低估）。
