# Code Review Checklist (8 大类)

> 评审 AI 在 Step 5 加载本检查清单。每个类别 3-5 个核心检查点。

## 1. 安全性 (Security)

- [ ] **硬编码密钥**：检查是否包含 API key / token / password 明文
- [ ] **SQL 注入**：用户输入是否走参数化查询
- [ ] **命令注入**：是否使用 `exec` / `eval` / 拼接 shell
- [ ] **XSS**：输出是否做 HTML 转义
- [ ] **路径遍历**：文件路径是否校验 `..` 越界

## 2. 正确性 (Correctness)

- [ ] **边界条件**：空数组、null、0、负数、最大值
- [ ] **并发安全**：共享状态是否有锁 / 原子操作
- [ ] **资源生命周期**：句柄、连接、订阅是否正确关闭
- [ ] **Off-by-one**：循环边界、数组下标
- [ ] **类型转换**：隐式转换、精度丢失、整数溢出

## 3. 性能 (Performance)

- [ ] **N+1 查询**：循环里是否有可批量的数据库调用
- [ ] **重复计算**：可缓存的昂贵操作
- [ ] **内存泄漏**：长生命周期对象持有短生命周期引用
- [ ] **算法复杂度**：O(n²) 是否可优化
- [ ] **I/O 同步阻塞**：是否阻塞主线程 / event loop

## 4. 可读性 (Readability)

- [ ] **命名清晰**：变量/函数/类名能否自解释
- [ ] **函数长度**：是否超过 30 行（建议拆）
- [ ] **嵌套深度**：是否超过 3 层（建议提前 return）
- [ ] **魔法数字**：硬编码常量是否提取为命名常量
- [ ] **注释质量**：是否解释"为什么"而非"是什么"

## 5. 可测试性 (Testability)

- [ ] **单元测试覆盖**：核心逻辑是否有测试
- [ ] **边界用例**：是否覆盖空、错、极端值
- [ ] **Mock 友好**：是否依赖具体实现（难以 mock）
- [ ] **测试隔离**：是否依赖全局状态 / 外部资源
- [ ] **断言质量**：是否检查关键副作用而非仅检查返回值

## 6. 错误处理 (Error Handling)

- [ ] **错误传播**：异常是否被吞掉 / 错误码是否被忽略
- [ ] **错误信息**：是否包含足够的诊断上下文
- [ ] **重试逻辑**：瞬时错误是否有退避重试
- [ ] **降级方案**：外部依赖失败时是否有 fallback
- [ ] **日志记录**：错误是否有可检索的日志

## 7. API 设计 (API Design)

- [ ] **向后兼容**：是否破坏现有调用方
- [ ] **命名一致性**：是否与项目其他 API 风格一致
- [ ] **入参校验**：是否在边界做防御性检查
- [ ] **返回结构**：成功/失败是否对称，错误码是否明确
- [ ] **幂等性**：重复调用是否产生相同结果

## 8. 文档 (Documentation)

- [ ] **公共 API 文档**：导出函数/类是否有 docstring / JSDoc
- [ ] **变更说明**：CHANGELOG / release notes 是否更新
- [ ] **架构决策**：重大改动是否有 ADR 记录
- [ ] **使用示例**：复杂 API 是否有示例
- [ ] **README 同步**：新增功能是否反映在 README
