# Runbook: High Error Rate（错误率飙升）

> 触发关键词：5xx 飙升 / 错误率 / error rate / 大量 5xx
> 典型严重度：P0 (>50%) / P1 (5-50%)

## 症状

- 5xx 错误率超过阈值（>5% 触发 P1，>50% 触发 P0）
- Sentry 报错激增
- 用户报告"接口报错"

## 排查步骤

1. **确认错误率**：`scripts/fetch-metrics.py <service> 5m` 看 error_rate
2. **错误类型分布**：查 5xx 占比（500/502/503/504 哪个多）
3. **查看日志**：`scripts/fetch-logs.py <service> ERROR` 拉错误日志
4. **查看 trace**：在 APM 中看慢调用 / 异常调用
5. **对比基线**：和昨天同时段 / 上周同时段对比

## 常见错误码诊断

| 错误码 | 含义 | 常见根因 |
|--------|------|---------|
| 500 | Internal Server Error | 代码异常 / 未捕获错误 |
| 502 | Bad Gateway | 上游服务不可用 |
| 503 | Service Unavailable | 过载 / 限流 / 维护 |
| 504 | Gateway Timeout | 上游超时 |

## 常见根因

| 根因 | 占比 | 排查信号 |
|------|------|---------|
| 最近部署引入 bug | 35% | deploy 时间与错误率上升时间吻合 |
| 数据库慢查询 | 20% | DB CPU 高 / 慢查询日志 |
| 上下游服务异常 | 15% | 上游同时告警 |
| 资源耗尽 | 15% | CPU/内存/连接池满 |
| 流量激增 | 10% | QPS 突增 |
| 缓存失效 | 5% | 缓存命中率下降 |

## 修复动作

### 按根因修复

1. **最近部署引入** → 立即回滚
2. **DB 慢查询** → kill 慢查询 / 加索引 / 限流
3. **上下游异常** → 联系对应服务 oncall
4. **资源耗尽** → 扩容 / 重启 / 调高连接池
5. **流量激增** → 限流 / 启用降级
6. **缓存失效** → 预热缓存 / 检查缓存服务

## 验证

1. 错误率回到 < 1%
2. 各错误码占比回到基线
3. 无新错误日志
4. 持续观察 30min

## 升级

- 错误率 > 50% 持续 5min → 升级 oncall 主管
- 涉及支付 / 资金 → 直接 P0 + 升级 VP
