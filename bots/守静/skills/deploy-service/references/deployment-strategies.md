# 部署策略对比

> 蓝绿 / 滚动 / 金丝雀三大策略详细对比与选择决策树。

## 策略矩阵

| 维度 | 蓝绿 (Blue-Green) | 滚动 (Rolling) | 金丝雀 (Canary) |
|------|------------------|---------------|----------------|
| **切换时间** | 立即（秒级） | 渐进（分钟级） | 渐进（10min+） |
| **资源占用** | 2×（双倍） | 1×（原样） | 1.1-1.5×（少量） |
| **回滚速度** | 秒级 | 分钟级 | 秒级（流量切回） |
| **风险** | 高（瞬时切换） | 中（分批） | 低（少量用户） |
| **DB schema 兼容** | 需双写窗口 | 需前向兼容 | 需前向兼容 |
| **用户影响** | 一次性 | 渐进 | 极小 |

## 适用场景

### 蓝绿 (Blue-Green)
- 重大版本变更（如 1.x → 2.x）
- DB schema 不兼容变更（需双写窗口）
- 紧急修复需秒级切换
- **不适用**：资源紧张环境、变更频繁

### 滚动 (Rolling，默认)
- 普通迭代发布
- 资源利用率敏感
- 无状态服务
- **不适用**：启动慢/有状态服务

### 金丝雀 (Canary)
- 灰度验证（先小流量）
- 风险敏感（金融/医疗）
- 性能对比（先 10% 流量）
- **不适用**：流量太小（无法形成统计意义）

## 选择决策树

```
是否涉及 DB schema 变更？
├─ 是 → 双写窗口可用？
│       ├─ 是 → 蓝绿
│       └─ 否 → 拒绝部署（升级 DBA 审核）
└─ 否 → 变更风险等级？
        ├─ 重大版本 → 蓝绿
        ├─ 中等变更 → 滚动
        └─ 风险敏感 → 金丝雀
```

## 流量切换流程

### 蓝绿
1. 部署 Green 环境（与 Blue 并存）
2. 健康检查通过
3. Service selector 切换 Green（蓝停接受流量）
4. 观察 5min
5. 保留 Blue 7 天后清理

### 滚动
1. 创建新 ReplicaSet
2. 逐步增加新 ReplicaSet 副本（1 → 1/2 → 全部）
3. 逐步减少旧 ReplicaSet 副本
4. 健康检查通过才继续

### 金丝雀
1. 部署 Canary 版本（10% 流量）
2. 观察 5min 业务指标
3. 通过 → 扩大至 50%
4. 观察 5min
5. 通过 → 100% 全量
6. 任一阶段失败 → 切回 100% 旧版本
