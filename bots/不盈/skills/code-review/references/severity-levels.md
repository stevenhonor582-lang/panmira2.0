# 严重度级别定义 (P0 / P1 / P2 / P3)

> 评审 AI 在 Step 5 应用本分级标准。每个发现必须标注严重级别。

## 严重度总览

| 级别 | 名称 | 响应时间 | 是否阻断合并 |
|------|------|---------|------------|
| P0 | 阻断 (Blocker) | 立即 | **是** — 必须修复 |
| P1 | 严重 (Critical) | 24h | **是** — 合并前修复 |
| P2 | 中等 (Major) | 1 周 | 否 — 建议修复 |
| P3 | 轻微 (Minor) | 下次迭代 | 否 — 可选修复 |
| INFO | 提示 (Info) | - | 否 — 仅记录 |

## P0 — 阻断 (Blocker)

**定义**：必须立即修复才能合并。

**典型场景**：
- 安全漏洞：硬编码密钥、SQL 注入、命令注入、认证绕过
- 数据丢失：未持久化关键数据、误删数据
- 资金/合规风险
- 系统级崩溃：影响生产环境

**评审输出**：
- 必改项（must_fix）
- 飞书紧急通知

## P1 — 严重 (Critical)

**定义**：合并前必须修复。

**典型场景**：
- 严重性能问题（响应时间 > 10x）
- 主要功能错误（影响主流程）
- 缺少关键错误处理（可能引发 P0 隐患）
- 公开 API 的破坏性变更无 deprecation

**评审输出**：
- 必改项（must_fix）

## P2 — 中等 (Major)

**定义**：建议在下个迭代修复。

**典型场景**：
- 性能优化（响应时间 < 2x 影响）
- 可读性问题影响维护
- 缺少测试覆盖核心逻辑
- 错误处理不完善（无 P0 隐患）

**评审输出**：
- 必改项（must_fix）或 建议项（suggestions），视项目阶段

## P3 — 轻微 (Minor)

**定义**：可选修复，不影响功能。

**典型场景**：
- 代码风格（命名、注释）
- 非关键路径的优化
- 文档不完善
- 重构机会

**评审输出**：
- 建议项（suggestions）

## INFO — 提示 (Info)

**定义**：仅记录，无修复要求。

**典型场景**：
- 引用了已知的 TODO
- 与其他模块风格不一致（但可接受）
- 提示性建议（如"可考虑 X"）

**评审输出**：
- 建议项（suggestions），severity=INFO

## 严重度判断辅助

### 升级标准

发现具有以下任一情况时，严重度 +1：
- 涉及公开 API / 对外服务
- 处于热点代码路径（高频调用）
- 修复成本 < 1h

### 降级标准

发现具有以下任一情况时，严重度 -1（最低 INFO）：
- 仅内部工具使用
- 处于实验性 / 弃用路径
- 已存在 > 6 个月无影响
